# Требования к внешним интеграциям (Умный дом)

## 1. Общие положения

### 1.1. Цель документа
Документ определяет требования к безопасности, протоколам взаимодействия и организации интеграции между системами PropDevelopment и внешней платформой партнёра, предоставляющей сервисы «Умный дом»:

*   Интеллектуальный домофон (биометрическое распознавание лиц, открытие по звонку)
*   Интеллектуальный шлагбаум (распознавание автомобильных номеров)

### 1.2. Участники интеграции

| Сторона | Роль | Системы |
| :--- | :--- | :--- |
| **PropDevelopment** | Заказчик сервисов | tenant-core-app, smart-home-integration-service, crm-tenant-db, AD/Keycloak |
| **Партнёр (Платформа Умного дома)** | Поставщик сервисов | API шлюз, устройства домофонов и шлагбаумов |

## 2. Требования к безопасности

### 2.1. Конфиденциальность (Confidentiality)

| № | Требование | Обоснование |
| :--- | :--- | :--- |
| **C1** | Все данные, передаваемые между PropDevelopment и платформой партнёра, должны шифроваться с использованием TLS 1.3 (или выше). | Предотвращение перехвата команд на открытие и персональных данных. |
| **C2** | Биометрические данные (шаблоны лиц) не должны передаваться в PropDevelopment. Партнёр хранит их на своей стороне. PropDevelopment передаёт только идентификаторы пользователей. | Минимизация рисков утечки биометрии. |
| **C3** | Номера автомобилей при передаче должны шифроваться, а храниться в хешированном виде. | 152-ФЗ (персональные данные). Номер автомобиля может быть косвенным идентификатором. |
| **C4** | Журналы событий (кто, когда, через какой доступ вошёл) должны передаваться по защищённому каналу. | Аудит инцидентов. |

### 2.2. Целостность (Integrity)

| № | Требование | Обоснование |
| :--- | :--- | :--- |
| **I1** | Все API-запросы должны быть подписаны с использованием SHA256 или цифровых сертификатов (mTLS). | Защита от подмены команд (например, ложная команда на открытие всех шлагбаумов). |
| **I2** | Списки доступа (белые списки номеров автомобилей, разрешённые лица) должны передаваться с контрольной суммой. | Проверка целостности при синхронизации. |
| **I3** | Платформа партнёра должна подтверждать получение и успешное применение изменений через эхо-запросы или webhook-подтверждения. | Гарантия, что доступ синхронизирован. |

### 2.3. Доступность (Availability)

| № | Требование | Обоснование |
| :--- | :--- | :--- |
| **A1** | Время ответа API партнёра на команду открытия должно быть не более 300 мс (p99). | Пользовательский опыт (домофон не должен задерживать). |
| **A2** | Партнёр должен обеспечить отказоустойчивую инфраструктуру (как минимум 2 дата-центра) и SLA доступности 99.9%. | Критичность сервиса (доступ в дом). |
| **A3** | При недоступности платформы партнёра smart-home-integration-service должна: (а) кэшировать команды на открытие на 5 минут, (б) уведомлять администратора. | Предотвращение полной потери функции. |
| **A4** | Должен быть реализован fallback-режим: классическое открытие по звонку (без биометрии) при сбоях API. | Безопасность (человек не должен остаться на улице). |

### 2.4. Неотказуемость (Non-repudiation)

| № | Требование | Обоснование |
| :--- | :--- | :--- |
| **N1** | Все действия по открытию доступа должны логироваться на обеих сторонах с метками времени (синхронизированными через NTP). | Расследование инцидентов (кто и когда открыл дверь). |
| **N2** | Логи должны храниться не менее 1 года. PropDevelopment имеет право запрашивать выгрузку логов. | Соответствие законодательству РФ. |

### 2.5. Tenant isolation (изоляция данных разных управляющих компаний)

| № | Требование | Обоснование |
| :--- | :--- | :--- |
| **T1** | Платформа партнёра должна поддерживать tenant isolation на уровне API. Каждый запрос от PropDevelopment должен содержать `X-Tenant-ID`, и партнёр обязан проверять, что устройство (домофон/шлагбаум) относится к этому tenant'у. | Прямое требование из выявленных проблем (партнёр из одной УК не должен видеть данные другой). |
| **T2** | PropDevelopment передаёт в партнёрскую систему только обезличенные идентификаторы квартир, собственников и устройств. | Дополнительный уровень изоляции. |

## 3. Протоколы аутентификации и авторизации

### 3.1. Аутентификация системы (machine-to-machine)

| Протокол | Способ применения | Требование |
| :--- | :--- | :--- |
| **mTLS (Mutual TLS)** | Используется для всех запросов между smart-home-integration-service и API партнёра. | Обязателен. Каждая сторона предъявляет валидный сертификат, выпущенный доверенным центром. |
| **Client Certificate** | Сертификат PropDevelopment хранится в защищённом хранилище (HSM или KMS). | Срок действия сертификата — не более 1 года. Автоматическое обновление. |

### 3.2. Аутентификация пользователя (через PropDevelopment)

| Протокол | Способ применения | Требование |
| :--- | :--- | :--- |
| **OAuth 2.0 / OpenID Connect** | Используется для аутентификации собственника и менеджера УК в мобильном приложении. | PropDevelopment предоставляет токен, который передаётся в smart-home-integration-service. |
| **JWT (JSON Web Token)** | Внутренний токен между tenant-core-app и smart-home-integration-service. | Содержит `tenant_id`, `user_id`, `role` (OWNER, UK_MANAGER). Время жизни — 15 минут. |

### 3.3. Авторизация (проверка прав)

| Механизм | Описание | Требование |
| :--- | :--- | :--- |
| **RBAC (Role-Based Access Control)** | tenant-core-app проверяет роль пользователя (OWNER / UK_MANAGER) и его право на управление конкретным домофоном/шлагбаумом. | Только OWNER своей квартиры или UK_MANAGER своего ЖК может изменять списки доступа. |
| **Policy-based access** | smart-home-integration-service проверяет, что команда на открытие пришла от авторизованного пользователя через tenant-core-app. | Запрет прямых вызовов API партнёра из мобильного приложения. |

### 3.4. Итоговая таблица протоколов

| Сценарий | Аутентификация | Авторизация | Шифрование |
| :--- | :--- | :--- | :--- |
| Мобильное приложение → tenant-core-app | OAuth 2.0 (Keycloak) | JWT (roles) | TLS 1.3 |
| tenant-core-app → smart-home-integration-service | JWT (служебный) | RBAC | TLS 1.3 + mTLS (опционально) |
| smart-home-integration-service → API партнёра | mTLS (сертификаты) | X-Tenant-ID | TLS 1.3 + mTLS |
| API партнёра → устройство (домофон/шлагбаум) | По согласованию (обычно API key) | Tenant ID из запроса | TLS 1.3 |

## 4. Организация взаимодействия между системами

### 4.1. Общая архитектура взаимодействия

```text
┌─────────────────────────────────────────────────────────────────────────────┐
│                           PropDevelopment                                    │
│  ┌──────────────┐    ┌─────────────────────┐    ┌────────────────────────┐  │
│  │   Мобильное  │───▶│   tenant-core-app   │───▶│ smart-home-integration │  │
│  │ приложение   │    │ (бизнес-логика, RBAC)│    │      -service          │  │
│  └──────────────┘    └─────────────────────┘    └───────────┬────────────┘  │
│                               │                            │              │
│                          ┌────▼────┐                       │              │
│                          │crm-     │                       │              │
│                          │tenant-db│                       │              │
│                          └─────────┘                       │              │
└────────────────────────────────────────────────────────────┼──────────────┘
                                                             │
                                                             │ mTLS + TLS 1.3
                                                             │
                                                     ┌───────▼───────────────┐
                                                     │   Платформа партнёра   │
                                                     │   (API Умного дома)    │
                                                     └───────────┬───────────┘
                                                                 │
                                                                 │ (закрытые сети)
                                                         ┌───────▼───────┐
                                                         │  Домофон      │
                                                         │  Шлагбаум     │
                                                         └───────────────┘
```


| Направление | Тип | Протокол | Описание |
| :--- | :--- | :--- | :--- |
| PropDevelopment → Партнёр | Синхронный | REST / mTLS | Отправка команд на открытие, синхронизация списков доступа |
| Партнёр → PropDevelopment | Асинхронный | Webhook (HTTP POST) | Уведомления о звонках и событиях доступа |
| Адаптер ←→ Партнёр | Оба направления | mTLS + TLS 1.3 | Все вызовы защищены взаимной аутентификацией |


## 4.2. Ключевые сценарии

### 4.2.1. Синхронизация списков доступа (только push)

1.  **Менеджер УК** добавляет номер авто / лицо.
2.  **tenant-core-app** → сохранение в `tenant-core-db` (источник истины).
3.  **tenant-core-app** → `smart-home-integration-service`.
4.  **Адаптер** → **API партнёра** (mTLS) → обновление белого списка.
5.  **Партнёр** → webhook-подтверждение → **Адаптер** → обновление статуса в БД.

### 4.2.2. Открытие по звонку в домофон

1.  **Посетитель** звонит в домофон.
2.  **Партнёр** → webhook (звонок) → **Адаптер**.
3.  **Адаптер** → `tenant-core-app` (поиск собственника по tenant и квартире).
4.  **Адаптер** → push-уведомление → **Мобильное приложение** собственника.
5.  **Собственник** нажимает «открыть» → `tenant-core-app` → **Адаптер** → **API партнёра**.
6.  **Партнёр** открывает домофон → webhook подтверждения.

### 4.2.3. Автоматическое открытие по лицу/номеру

1.  **Партнёр** распознаёт лицо/номер.
2.  Проверка по своему белому списку (синхронизированному).
3.  Открытие двери/шлагбаума.
4.  **Партнёр** → webhook (событие: кто, когда, где) → **Адаптер**.
5.  **Адаптер** → сохранение события в `tenant-core-db` → push собственнику.
